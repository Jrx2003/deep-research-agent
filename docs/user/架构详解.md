# 架构详解

## 为什么这样设计？

### 多Agent vs 单Agent

**为什么不使用一个强大的Agent完成所有任务？**

1. **成本优化**：不同任务需要不同的推理能力
   - 搜索查询生成 → 轻量模型足够
   - 质量审查 → 需要强模型
   - 单Agent只能全部用强模型，成本高昂

2. **可观测性**：每个Agent有明确的输入输出
   - 便于调试和问题定位
   - 可以单独优化每个环节

3. **可复用性**：Agent可以独立升级或替换
   - 可以替换Planner实现不同的规划策略
   - 可以添加新的Researcher支持不同的数据源

### LangGraph vs 自定义框架

**为什么选择LangGraph？**

| 特性 | LangGraph | 自定义框架 |
|------|-----------|------------|
| 开发速度 | 快（内置状态管理、并行执行） | 慢（需从零实现） |
| 生态兼容 | 与LangChain无缝集成 | 需自行适配 |
| 可视化 | 内置Mermaid图生成 | 需自行实现 |
| 学习成本 | 中等 | 高（内部实现复杂） |
| 灵活性 | 足够 | 完全可控 |

对于本项目，LangGraph提供了足够的灵活性，同时大幅降低了开发成本。

## Agent设计详解

### 1. Planner Agent

**职责**：将研究主题分解为可执行的子查询

**输入**：
```python
{
    "query": "量子计算最新进展",
    "context": null  # 可选的额外上下文
}
```

**输出**：
```python
{
    "strategy": "分领域调研",
    "sub_queries": [
        "量子计算硬件最新突破 2024",
        "量子算法在AI中的应用",
        "量子计算商业应用案例",
        "主要量子计算公司进展"
    ],
    "expected_sections": ["硬件", "算法", "应用", "市场"]
}
```

**模型选择**：Medium tier (GPT-4o / Claude Sonnet)
- 需要一定的推理能力来分解问题
- 但不需要最强模型（有Critic后续审查）

### 2. Researcher Agent

**职责**：执行搜索，收集原始信息

**设计决策**：

**为什么批量搜索而非流式？**
- 批量搜索可以并行执行，效率更高
- 便于后续去重和排序
- 可以统一处理搜索结果

**搜索策略**：
```python
# 每个子查询执行多源搜索
for query in plan.sub_queries:
    # 1. DuckDuckGo搜索（免费，无限制）
    # 2. 如有SerpAPI，补充Google结果
    # 3. 去重和排序
    # 4. 抓取前N个结果的内容
```

**输出**：
```python
{
    "findings": [
        {
            "query": "量子计算硬件最新突破 2024",
            "sources": [
                {
                    "url": "...",
                    "title": "...",
                    "content": "...",
                    "relevance_score": 0.92
                }
            ]
        }
    ]
}
```

**模型选择**：Light tier (GPT-4o-mini / Claude Haiku)
- 主要工作是调用搜索工具
- 简单的内容提取和格式化

### 3. Synthesis Agent

**职责**：整合研究结果，形成连贯的章节

**设计挑战**：如何处理冲突信息？

**解决方案**：
```python
# 在合成时标注信息来源和置信度
content = """
量子计算硬件在2024年取得了多项突破。

IBM发布了Condor处理器，拥有1121个量子比特[来源1, 置信度: 高]，
但Google的研究指出量子纠错仍是主要挑战[来源2, 置信度: 高]。

有报道称某初创公司实现了1000量子比特[来源3, 置信度: 低]，
但尚未经过同行评审。
"""
```

**模型选择**：Medium tier (GPT-4o / Claude Sonnet)
- 需要理解和整合多个来源
- 需要一定的写作能力

### 4. Critic Agent

**职责**：审查研究质量，识别缺失和错误

**这是关键的质量控制环节**

**审查维度**：
```python
review_criteria = [
    "completeness",    # 是否覆盖了所有关键方面
    "accuracy",        # 事实是否准确
    "relevance",       # 内容是否切题
    "sources",         # 引用是否充分可靠
    "balance",         # 观点是否平衡
]
```

**输出**：
```python
{
    "passed": False,
    "score": 0.75,
    "feedback": [
        {
            "dimension": "completeness",
            "issue": "缺少中国量子计算公司的进展",
            "severity": "high",
            "suggestion": "添加对国盾量子、本源量子等的调研"
        }
    ],
    "revised_plan": {
        "additional_queries": ["中国量子计算公司 2024"]
    }
}
```

**模型选择**：Strong tier (o1 / Claude Opus)
- 需要最强的推理能力来发现缺陷
- 这是质量保证的关键环节，值得投入

### 5. Writer Agent

**职责**：生成最终的 polished 报告

**设计决策**：

**为什么不直接输出Synthesis的结果？**
- Synthesis的结果是结构化的内部表示
- Writer负责转换为最终的、适合阅读的格式
- 可以应用模板和样式

**模板系统**：
```python
templates = {
    "research_report": "标准研究报告",
    "literature_review": "文献综述",
    "competitor_analysis": "竞品分析",
    "technical_survey": "技术调研",
}
```

**模型选择**：Strong tier (o1 / Claude Opus)
- 需要优秀的写作能力
- 这是最终输出，直接影响用户体验

## 反馈循环设计

### 为什么要循环？

单次通过的问题：
1. 初始规划可能不完善
2. 搜索结果可能不充分
3. 需要迭代来收敛到高质量结果

### 循环终止条件

```python
def should_continue(state: ResearchState) -> bool:
    # 条件1：达到最大迭代次数
    if state.iteration >= MAX_ITERATIONS:
        return False

    # 条件2：Critic通过
    if state.review and state.review.passed:
        return False

    # 条件3：改进幅度太小（收敛）
    if state.review and state.review.score > 0.9:
        return False

    return True
```

### 循环优化策略

每次循环不是从头开始，而是增量改进：
```python
def planner_with_feedback(state: ResearchState):
    if state.review:
        # 基于反馈调整计划
        prompt = f"""
        原查询: {state.query}
        上一轮计划: {state.plan}
        Critic反馈: {state.review.feedback}

        请生成改进的查询计划，重点关注缺失的部分。
        """
    else:
        # 首次规划
        prompt = f"""
        查询: {state.query}
        请生成完整的查询计划。
        """
```

## 状态管理设计

### 为什么使用TypedDict？

LangGraph推荐使用TypedDict作为状态类型：

```python
from typing import TypedDict, Annotated
from operator import add

class ResearchState(TypedDict):
    # Annotated定义reducer函数
    findings: Annotated[list, add]  # 使用add合并列表
    cost: CostTracker  # 覆盖更新
```

### 状态字段设计

```python
class ResearchState(TypedDict):
    # === 输入 ===
    query: str                          # 原始查询
    context: Optional[str]             # 额外上下文
    config: ResearchConfig             # 研究配置

    # === 执行过程 ===
    iteration: int                     # 当前迭代次数
    plan: Optional[ResearchPlan]       # 查询计划
    findings: Annotated[list, add]     # 研究发现（累积）
    sections: List[Section]            # 合成后的章节
    review: Optional[ReviewResult]     # 审查结果

    # === 输出 ===
    report: Optional[str]              # 最终报告
    sources: List[Source]              # 引用来源

    # === 元数据 ===
    trace_id: str                      # 追踪ID
    cost: CostTracker                  # 成本追踪
    logs: List[LogEntry]               # 执行日志
```

## 模型路由策略

### 路由决策表

| 任务类型 | 模型 | 理由 |
|----------|------|------|
| 查询规划 | GPT-4o | 需要推理能力，但单次调用 |
| 批量搜索 | GPT-4o-mini | 重复任务，成本控制 |
| 内容提取 | GPT-4o-mini | 结构化输出，无需复杂推理 |
| 信息合成 | GPT-4o | 需要理解和整合多源信息 |
| 质量审查 | o1-preview | 需要最强推理发现缺陷 |
| 报告撰写 | Claude Opus | 需要优秀的写作能力 |

### 动态路由

基于任务复杂度动态选择：

```python
def route_by_complexity(task: Task) -> str:
    if task.estimated_tokens < 1000:
        return "gpt-4o-mini"
    elif task.estimated_tokens < 4000:
        return "gpt-4o"
    else:
        return "o1-preview"
```

## 可观测性设计

### 多层次追踪

```
Research Session (trace)
├── Planner Node (span)
│   ├── LLM Call (span)
│   └── Parse Output (span)
├── Researcher Node (span)
│   ├── Search 1 (span)
│   ├── Search 2 (span)
│   └── LLM Call (span)
├── Synthesis Node (span)
│   └── LLM Call (span)
├── Critic Node (span)
│   └── LLM Call (span)
└── Writer Node (span)
    └── LLM Call (span)
```

## 扩展性设计

### 添加新Agent

```python
# 1. 实现Agent函数
def my_new_agent(state: ResearchState) -> ResearchState:
    ...

# 2. 注册到Graph
graph.add_node("my_agent", my_new_agent)

graph.add_edge("previous_agent", "my_agent")
```
